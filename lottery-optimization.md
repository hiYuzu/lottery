# 🎉 多人同步抽奖系统 — 优化与新功能建议

## 🔧 可以优化的地方

### 1. 安全性（比较重要）
- **管理操作无鉴权**：目前任何连接 WebSocket 的人都可以发送 `reset`、`updatePrizes` 消息，没有身份验证。建议增加管理员密码或 token 机制
- **XSS 防护**：虽然前端有 `escapeHtml`，但服务端对奖项名称等字段没有做输入过滤，恶意内容可能通过 WebSocket 注入

### 2. 代码重复
- `escapeHtml`、`showToast`、WebSocket 连接/重连逻辑、名单加载逻辑在 `script.js` 和 `admin.html` 中重复出现。可以抽成一个公共的 `common.js`
- `admin.html` 的 JS 内联在 HTML 中，与 `index.html` 分离 JS 的风格不一致

### 3. 前端架构
- `script.js` 有 621 行，所有逻辑都在一个 IIFE 里。如果项目继续迭代，可以拆分为模块（如 `ws.js`、`animation.js`、`render.js`）
- 没有加载状态（loading）和错误边界处理，比如名单加载失败后用户可能误操作

### 4. 状态管理
- `JSON.parse(JSON.stringify(cachedState))` 做深拷贝在小数据量下没问题，但状态变大后会有性能开销
- 互斥锁是基于单进程内存的，如果将来需要多实例部署就不适用了

### 5. 移动端体验
- 抽奖按钮 `btn-draw-action` 固定 `padding: 14px 48px`，在小屏幕上可能偏大
- 滚动名字 `font-size: 48px` 在手机上可能溢出

### 6. 小细节
- WebSocket 断线重连期间，用户点击抽奖按钮只会提示"未连接"，可以改为自动排队等重连后发送
- 烟花动画的 `ctx.fillStyle = 'rgba(0, 0, 0, 0.15)'` 会导致画面残留淡影，可以用 `clearRect` + 自行管理粒子生命周期来实现更干净的消散效果
- 没有配置 lint 和测试框架

## ✨ 可以增加的功能

### 实用型功能

| 功能 | 说明 |
|------|------|
| **撤销上一次抽奖** | 误操作时可以回退，恢复中奖者到候选人池 |
| **大屏展示模式** | 专门适配投影仪/大屏的全屏展示页面，字号更大、动画更炫 |
| **二维码入口** | 生成局域网访问二维码，方便手机扫码参与 |
| **导出为 Excel** | 目前只支持 `.txt`，增加 `.xlsx` 导出更实用 |
| **抽奖倒计时** | 点击开始后 3-2-1 倒计时再滚动，增加仪式感 |

### 增强型功能

| 功能 | 说明 |
|------|------|
| **管理员登录** | 密码鉴权，普通用户只能观看/抽奖，管理员才能修改配置和重置 |
| **多轮抽奖会话** | 支持创建独立的抽奖活动，保留历史活动记录 |
| **奖项图片** | 每个奖项可以配一张奖品图片，抽奖时展示 |
| **动画主题** | 提供多种抽奖动画风格（经典滚动、转盘、卡片翻转等） |
| **实时在线人数** | 显示当前有多少客户端连接 |
| **权重抽奖** | 给不同人员设置不同中奖权重 |

### 体验增强

| 功能 | 说明 |
|------|------|
| **音效自定义上传** | 允许管理员上传自定义背景音乐和中奖音效 |
| **中奖结果分享图** | 生成带背景的中奖海报图片，方便分享 |
| **快捷键支持** | 空格键开始/停止抽奖，ESC 关闭弹窗 |
| **名单管理后台** | 目前名单需要手动编辑 JSON，可以在管理后台直接增删改人员 |
